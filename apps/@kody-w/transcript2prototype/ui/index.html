Transcript2Prototype

Paste a business transcript into your brainstem chat; walk out with a demoed, tested, exported agent prototype. One cubby per prototype. One file to install.
1 Demo script + M365 iframe bytecode 2 Conversational adjust 3 Real agent.py build 4 Local twin run 5 Live twin run 6 Factory singleton export (GATE)

Install

curl -fsSL https://raw.githubusercontent.com/kody-w/RAPP_Store/main/apps/@kody-w/transcript2prototype/singleton/transcript2prototype_agent.py \
  -o ~/.brainstem/src/rapp_brainstem/agents/transcript2prototype_agent.py

No restart needed - the brainstem hot-loads agents on every request. Open your brainstem chat and say: Build a prototype from this transcript: ...

What it does

  1. Start - your copilot analyzes the transcript and injects 3-5 capabilities as a parameter; the agent generates a turn-by-turn demo script, injects it into a static M365 Copilot demo page, base64-encodes it, and surfaces it as iframe bytecode in a rapplication shell HTML. Drive it with the Up arrow and Enter.
  2. Adjust - ask for changes in plain language; your copilot rewrites the turns through structured calls and the bytecode regenerates in place, at any stage.
  3. Build - one real agent.py per capability lands in the prototype's cubby.
  4. Test - the same demo script replays automatically against the generated agents, first in-process (local twin), then injected into your live brainstem over HTTP. On pass, the iframe flips to live mode: the demo you rehearsed now drives the real agents.
  5. Export - everything bundles into one factory singleton agent.py. The pipeline stops there by design: that file is the hand-off for the next stage (customer sandbox, Copilot Studio packaging, partner).
The caller contract: nothing is hardcoded. The hosting LLM is the intelligence and every input is a parameter whose description teaches the caller exactly what to provide. Even the fallback analyzer's knobs (pain markers, capability vocabulary, max capabilities) are parameters.
